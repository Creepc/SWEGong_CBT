## 제1장 계획의 이해와 타당성 분석

자료: 3장_계획의 이해.pdf

### 주요 학습 주제
- 소프트웨어 개발 계획은 비용, 기간, 자원 계획을 세워 일정 지연, 비용 초과, 품질 저하, 유지보수 비용 증가를 예방하기 위한 활동이다.
- 문제 정의는 소프트웨어 개발의 첫 작업이며, 무엇을 개발할 것인지와 개발 범위를 명확히 하여 초기 타당성 분석과 초기 계획의 기초 자료가 된다.
- 타당성 분석은 경제적, 기술적, 법적 타당성으로 구분된다.
- 개발비 산정은 제조나 건축보다 어렵다. 사람 중심 개발, 개발자 능력 차이, 다양한 개발 프로세스, 품질·기간 영향이 복합적으로 작용하기 때문이다.
- 개발비에 영향을 주는 요소에는 프로그램 자질, 소프트웨어 복잡도, 소프트웨어 크기, 가용시간, 요구 신뢰도 수준, 기술 수준이 있다.
- 브룩스 법칙은 지체되는 프로젝트에 인력을 추가하면 신규 인력 교육과 의사소통 부담 때문에 오히려 개발이 늦어질 수 있음을 말한다.

### 계획의 의미

소프트웨어 개발 계획은 단순히 일정을 적어 놓는 문서가 아니다. 개발에 필요한 비용, 기간, 자원을 미리 예측하고 배분하는 활동이다. 계획이 없으면 일정 지연, 비용 초과, 품질 저하, 유지보수 비용 증가가 발생한다. 따라서 계획은 문제 정의, 타당성 분석, 개발비용 산정, 일정계획, 위험분석으로 이어지는 흐름으로 이해해야 한다.

시험에서는 계획의 필요성을 결과 중심으로 묻는 경우가 많다. '계획이 없으면 어떤 문제가 발생하는가'라는 질문에 일정, 비용, 품질, 유지보수 네 축을 함께 답할 수 있어야 한다.

### 문제 정의

문제 정의는 소프트웨어 개발의 첫 작업이다. 여기서는 무엇을 개발할 것인지 명확히 정의하고 개발 범위를 결정한다. 이 결과는 초기 타당성 분석과 초기 계획 작성의 기초 자료가 된다. 문제 정의를 잘하려면 개발하고자 하는 분야의 배경지식이 필요하고, 현재 운영 중인 시스템을 사용해 보며, 실무 담당자와 면담하여 자료를 수집하고 분석해야 한다.

함정 포인트: 문제 정의는 요구사항 명세 전체를 완성하는 단계가 아니라, 개발 대상과 범위를 명확히 잡아 이후 타당성 분석과 계획 수립의 근거를 만드는 단계이다.

### 타당성 분석

타당성 분석은 개발 여부를 판단하기 위한 검토 활동이다. 경제적 타당성은 투자 효율성, 비용-효익 분석, 시장성을 검토한다. 기술적 타당성은 현재 기술로 요구 기능을 구현할 수 있는지, 하드웨어 성능이 충분한지, 개발자의 기술력에 문제가 없는지 검토한다. 법적 타당성은 개발용 소프트웨어와 도구 사용이 소유권 등 법적 문제를 일으키지 않는지, 지적소유권과 프로그램 보호법에 저촉되지 않는지 검토한다.

경제적 타당성은 경영자 관점의 투자 효율성과 연결되고, 분석가는 비용-효익 검토 결과를 경영자에게 정확히 제공해야 한다. 기술적 타당성은 '기술 자체가 존재하는가'만이 아니라 하드웨어와 개발자 역량까지 포함한다. 법적 타당성은 도구 사용과 지식재산권을 포함한다.

### 개발비 산정의 어려움

전자제품 생산이나 건축은 생산 제품 형상, 자재 수량, 자재 가격, 평당 가격과 평수처럼 비용 산정의 기준이 비교적 명확하다. 반면 소프트웨어 개발은 사람, 즉 개발자 중심이다. 개발자의 능력에 따라 생산성 차이가 크고, 이 차이는 기간과 품질에 영향을 준다. 또한 개발 프로세스가 다양하여 표준화와 자동화가 어렵고, 생산성과 품질도 달라진다.

시험에서는 제조 원가와 소프트웨어 개발비의 차이를 비교하는 방식으로 자주 출제될 수 있다. 핵심은 소프트웨어가 물리적 자재 중심이 아니라 사람과 지식 노동 중심이라는 점이다.

### 개발비 영향 요소

개발비용에 영향을 주는 요소는 여러 가지다. 프로그램 자질은 초급 프로그래머와 고급 프로그래머의 생산성 차이를 의미한다. 소프트웨어 복잡도는 개발 유형과 난이도에 따라 비용을 증가시킨다. 소프트웨어 크기가 커지면 참여 인력이 증가하고, 개발기간이 길어지며, 복잡도도 커진다. 가용시간은 단순히 인력과 자원을 늘린다고 항상 개발기간이 줄어드는 것이 아니라는 점과 연결된다. 요구되는 신뢰도 수준이 높으면 개발비용이 증가한다. 기술 수준 측면에서는 고급언어가 저급언어보다 5~10배 생산성을 높일 수 있다고 설명된다.

브룩스 법칙은 지체되는 프로젝트에 인력을 추가하는 것이 개발을 늦출 뿐이라는 문장으로 제시되었다. 신규 투입 인력 교육이 필요하고 의사소통 부담도 커지기 때문이다. 따라서 '인력을 늘리면 일정은 항상 단축된다'는 선지는 함정이다.

### 시험 함정 포인트

- **소프트웨어 개발 계획**: 일정 지연·비용 초과·품질 저하·유지보수 비용 증가를 막기 위한 출발점이다.
- **문제 정의**: 초기 타당성 분석과 초기 계획 작성의 기초 자료가 된다.
- **경제적 타당성**: 개발 여부 판단과 경영자 의사결정에 연결된다.
- **기술적 타당성**: 기능 구현 가능성뿐 아니라 성능과 인력 기술력도 포함한다.
- **법적 타당성**: 개발 가능해 보여도 법적 문제가 있으면 추진이 어렵다.
- **개발비 산정의 어려움**: 제조·건축처럼 자재 수량만으로 산정하기 어렵다.
- **프로그램 자질**: 개발비와 기간, 품질에 영향을 준다.
- **소프트웨어 복잡도**: 복잡도는 개발비용에 직접 영향을 준다.

## 제2장 비용산정 기초와 하향식·상향식 산정

자료: 3장_계획의 이해.pdf, 3장_비용산정기법.pdf, 소프트웨어비용산정기법.pdf

### 주요 학습 주제
- 하향식 산정은 전문가 판단과 델파이 기법처럼 전체 수준에서 비용을 산정한다.
- 상향식 산정은 세부 작업 단위별 비용을 먼저 산정하고 전체 비용을 합산한다.
- LOC 기법은 원시 코드 라인 수의 낙관치, 중간치, 비관치를 이용해 추정 LOC를 구하고 노력·비용·기간·생산성을 계산한다.
- 개발단계별 노력 기법은 코딩만 보는 LOC의 한계를 보완하여 생명주기 각 단계의 M/M을 산정한다.

### 하향식 산정

하향식 산정 기법은 전체 프로젝트 수준에서 비용을 먼저 산정하는 접근이다. 강의자료에서는 전문가 판단 기법과 델파이 기법이 제시된다. 전문가 판단 기법은 경험이 많은 전문가가 개발비용을 산정하기 때문에 짧은 시간에 개발비를 산정하거나 입찰에 응해야 하는 경우 많이 사용된다. 단점은 수학적 계산 방법보다 경험에만 의존할 경우 부정확할 수 있다는 점이다.

하향식은 빠르지만 세부 작업이 충분히 분해되지 않은 상태에서 경험에 기대는 경우가 많다. 그래서 시험에서는 '신뢰성이 높다'와 '경험에만 의존하면 부정확할 수 있다'를 함께 이해해야 한다.

### 상향식 산정

상향식 산정 기법은 세부 작업 단위별로 비용을 산정한 뒤 전체 비용을 합산하는 방식이다. 작업이 충분히 분해되어 있으면 직관적이고 빠르게 적용할 수 있지만, 수학적 계산보다 경험에만 의존하면 부정확할 수 있다는 단점은 여전히 존재한다. 강의자료에서는 상향식 산정의 대표로 LOC 기법과 개발단계별 노력 기법을 다룬다.

하향식과 상향식의 차이는 출발점이다. 하향식은 전체에서 내려오고, 상향식은 세부 작업에서 올라간다.

### LOC 기법

LOC 기법은 개발하려는 소프트웨어의 총 코드 라인 수를 예측하여 구현 단계에 대한 M/M을 산정하는 방법이다. 각 모듈의 낙관치, 중간치, 비관치를 추정하고, PERT 가중 평균 공식으로 추정 LOC를 계산한다. 공식은 추정 LOC = (낙관치 + 4×중간치 + 비관치) / 6 이다. 중간치에 가중치 4를 주는 이유는 가장 그럴듯한 값에 더 큰 비중을 두기 위해서다.

추정 LOC가 나오면 1인당 월평균 생산 코드 라인 수로 나누어 노력(M/M)을 구한다. 그 다음 개발비용 = M/M × 1인당 월평균 인건비, 개발기간 = M/M / 참여인원, 생산성 = LOC / M/M 을 계산한다.

### M/M의 의미

M/M은 Man-Month 또는 Person-Month로, 한 사람이 한 달 일한 양을 기준으로 한 노력 단위다. 예를 들어 5명의 백엔드 개발자가 12개월 참여하고 7명의 프론트엔드 개발자가 5개월 참여하면 노력은 5×12 + 7×5 = 95 M/M이다. M/M은 기간 그 자체가 아니라 총 투입 노력이다. 100 M/M 프로젝트라도 10명이 참여하면 단순 계산상 10개월이지만, 실제 프로젝트에서는 의사소통과 작업 의존성 때문에 항상 선형적으로 줄어들지는 않는다.

### 개발단계별 노력 기법

LOC는 구현 단계, 특히 코딩을 중심으로 M/M을 산정한다. 그러나 실제 소프트웨어 개발은 코딩뿐 아니라 요구분석, 설계, 테스트 등 여러 단계에서 인력과 자원이 필요하다. 개발단계별 노력 기법은 M/M을 소프트웨어 개발 생명주기의 각 단계에 적용하여 단계별로 산정한다. 그래서 코딩만 대상으로 산정하는 LOC보다 더 정확할 수 있다.

### 시험 함정 포인트

- **하향식 산정**: 빠르지만 경험 의존성이 단점이다.
- **전문가 판단 기법**: 입찰 등 짧은 시간 산정에 사용된다.
- **델파이 기법**: 자료에서 하향식 산정의 항목으로 제시된다.
- **상향식 산정**: 작업분해가 되어 있을 때 직관적이다.
- **LOC 기법**: 구현 단계 근처에서 특히 사용된다.
- **낙관치**: PERT 공식에서 가중치 1로 들어간다.
- **중간치**: PERT 공식에서 가중치 4로 가장 크게 반영된다.
- **비관치**: PERT 공식에서 가중치 1로 들어간다.

## 제3장 수학적 비용산정: COCOMO, COCOMO II, 기능점수 개요

자료: 3장_계획의 이해.pdf, 3장_비용산정기법.pdf, 소프트웨어비용산정기법.pdf, 각종용어정리_노션.pdf

### 주요 학습 주제
- 수학적 산정 기법은 COCOMO, Putnam, 기능점수 방법을 포함한다.
- COCOMO는 LOC 중심의 개발비 산정 모델로 KDSI, 프로젝트 유형, EAF를 사용한다.
- COCOMO 프로젝트 유형은 단순형, 중간형, 내장형으로 나뉘며 복잡할수록 노력 증가율이 커진다.
- COCOMO II는 계획 단계에서 LOC를 미리 예측해야 하는 기존 COCOMO의 한계를 보완하기 위해 단계별 모델을 사용한다.
- 기능점수는 사용자 관점의 기능 개수와 복잡도로 소프트웨어 규모를 측정한다.

### 수학적 산정 기법의 위치

강의자료에서 수학적 산정 기법은 상향식 비용산정 기법이면서 경험적 또는 실험적 추정 기법으로 설명된다. 대표 기법은 COCOMO, Putnam, 기능점수 방법이다. COCOMO는 소프트웨어 규모를 LOC로 예측한 뒤 소프트웨어 종류에 따른 공식에 대입하여 비용을 산정한다. Putnam은 소프트웨어 생명주기의 전 과정에 사용될 노력의 분포를 가정하는 방법으로 제시된다. 기능점수 방법은 기능점수를 구한 후 이를 이용해 비용을 산정한다.

### COCOMO의 기본 구조

COCOMO는 Constructive Cost Model의 약자로 라인 수 중심의 개발비 산정 모델이다. 입력값은 KDSI이며, KDSI는 Kilo Delivered Source Instructions, 즉 납품되는 순수 코드 줄 수를 1000줄 단위로 측정한 값이다. 주석, 공백, 라이브러리 호출은 보통 제외된다. DSI가 10,000줄이면 KDSI는 10이다.

기본 공식은 PM = A × (KDSI)^B × EAF 이고, 총 개발기간은 TDEV = 2.5 × (PM)^β 로 계산한다. PM은 Person-Month, EAF는 Effort Adjustment Factor이다. A, B, β는 프로젝트 유형에 따라 달라진다.

### COCOMO 프로젝트 유형

단순형 프로젝트는 복잡도와 난이도가 비교적 높지 않은 업무용 소프트웨어로, 크기는 중소규모이며 50KDSI 이하로 설명된다. 중간형 프로젝트는 규모나 복잡도가 중간급이며 300KDSI 이하, 운영체제나 데이터베이스 관리 프로그램이 예로 제시된다. 내장형 프로젝트는 자동화기기나 전자제품처럼 하드웨어와 밀접하게 관련된 소프트웨어이며 크기는 300KDSI 이상으로 설명된다.

보충 교안의 계수표에서는 Organic(단순형) A=2.4, B=1.05, β=0.38, Semi-detached(중간형) A=3.0, B=1.12, β=0.35, Embedded(내장형) A=3.6, B=1.20, β=0.32로 정리되어 있다. B가 1보다 크므로 규모가 커질수록 노력은 비선형적으로 증가한다. 특히 Embedded의 B가 가장 커서 증가율이 가장 가파르다.

### EAF와 Cost Driver

EAF는 노력조정수치로, 보정에 사용되는 값이다. 제품, 컴퓨터, 인력, 프로젝트의 네 특성 영역에 대해 총 15개 Cost Driver 승수를 곱해서 구한다. 예를 들어 요구 신뢰도 높음 1.15, 복잡도 매우 높음 1.30, 현대적 개발방법 거의 사용 안 함 1.24, 일정 매우 촉박 1.10이면 EAF = 1.15×1.30×1.24×1.10 = 약 2.04가 된다.

함정 포인트: 모든 보정값이 1보다 큰 것은 아니다. 분석가 능력, 프로그래머 능력, 도구 사용, 현대적 개발방법 사용처럼 역량이나 도구가 좋을수록 승수가 1보다 작아져 노력을 줄인다. 반면 실행시간 제약이나 주기억 제약은 제약이 높을수록 1보다 커진다.

### COCOMO II

기존 COCOMO의 문제점은 계획 단계에서 원시 코드 라인 수를 미리 예측해야 한다는 점이다. COCOMO II는 프로젝트 진행 정도에 따라 값을 예측한 뒤 인건비를 예측한다. 1단계는 애플리케이션 합성 모델로 UI 개수와 객체 점수 등을 통해 규모를 산정한다. 2단계는 초기설계 모델로 초기 설계 단계에서 예측값을 구하며 1단계보다 정확하다. 3단계는 구조설계 이후 모델로 기능점수를 바탕으로 LOC를 추정하여 소프트웨어 규모를 산정한다.

각종 용어 정리 자료에서는 Post-Architecture Model을 구조 설계가 완료된 이후 소프트웨어 개발 노력을 예측하는 가장 정밀한 모델로 설명하며, 기능점수 또는 LOC와 비용 승수를 사용한다고 정리한다.

### 기능점수 개요

기능점수는 소프트웨어 기능의 크기를 측정하는 단위다. 사용자 관점에서 소프트웨어 기능이 얼마나 복잡한지를 상대적인 점수로 표현한다. 산정 근거는 입출력, 데이터베이스 테이블, 조회 등 기능의 수이다. 라인 수와 무관하게 기능이 많으면 규모가 크고 복잡도가 높다고 판단한다.

기능점수의 특징은 기능적 요구사항 중심, 사용자 관점의 정량화, 개발 기술·개발방법·품질수준과 무관한 측정, 개발 언어와 무관, 생명주기 전체 단계에서 사용 가능하다는 점이다.

### 시험 함정 포인트

- **수학적 산정 기법**: 경험적 또는 실험적 추정 기법으로 설명된다.
- **COCOMO**: LOC 중심 개발비 산정이다.
- **KDSI**: 주석·공백·라이브러리 호출은 보통 제외된다.
- **DSI**: DSI/1000 = KDSI이다.
- **PM**: COCOMO의 주 출력값이다.
- **TDEV**: PM에 유형별 지수를 적용한다.
- **단순형 Organic**: 50KDSI 이하로 설명된다.
- **중간형 Semi-detached**: 운영체제, DBMS 등이 예시다.

## 제4장 기능점수 상세 산정과 5가지 기능 유형

자료: 3장_계획의 이해.pdf, 3장_비용산정기법.pdf, 소프트웨어비용산정기법.pdf, 부록A_5가지 기능 유형 구분 기준.pdf

### 주요 학습 주제
- 기능점수 산정은 데이터 기능(ILF, EIF)과 트랜잭션 기능(EI, EO, EQ)을 구분하는 것이 핵심이다.
- 파일 기능은 유지보수 책임으로 ILF와 EIF를 구분한다.
- 트랜잭션 기능은 방향과 계산·가공 여부로 EI, EO, EQ를 구분한다.
- 간이 기능점수법은 초기 단계에서 평균복잡도 가중치를 사용한다.
- UFP와 보정 전·후 개발원가 산정 과정은 계산 문제로 출제되기 쉽다.

### 기능점수 산정의 큰 흐름

기능점수는 기능을 먼저 도출하고, 기능 유형을 분류한 뒤, 각 유형의 평균 가중치를 적용하여 기능점수를 계산한다. 정규기능점수법은 설계단계 이후에 사용하면 유용하고, 간이기능점수법은 기획 및 발주 단계에서 사용된다. 간이기능점수법은 프로젝트 초기 단계에서 평균복잡도 가중치를 사용하여 소프트웨어 기능의 크기를 측정한다.

### 측정유형과 경계

간이 기능점수 산정 절차의 첫 단계는 측정유형 결정이다. 개발 프로젝트 기능점수는 프로젝트가 완료되어 최초 설치했을 때 사용자에게 제공된 모든 기능의 크기를 측정한다. 개선 프로젝트 기능점수는 운영 중인 소프트웨어에 변경이 발생했을 때 추가, 수정, 삭제된 부분만 측정한다. 애플리케이션 기능점수는 현재 운용 중인 애플리케이션의 기능을 측정하며 개발 프로젝트 기능점수에 개선 프로젝트 기능점수까지 포함한다.

다음은 측정범위와 애플리케이션 경계 식별이다. 측정범위는 기능점수를 측정하고자 하는 대상이고, 애플리케이션 경계는 금번 기능점수를 계산하는 대상과 다른 애플리케이션이나 외부 사용자를 구분하는 경계다. 경계는 사용자 관점에서 구분해야 하며, 클라이언트/서버나 서버/비즈니스로직/클라이언트 같은 개발자 관점으로 구분하면 안 된다.

### 데이터 기능: ILF와 EIF

데이터 기능은 내부논리파일(ILF)과 외부연계파일(EIF)로 나뉜다. ILF는 사용자가 등록, 수정, 삭제, 조회하기 위한 대상이며, 애플리케이션 내부에서 파일로 유지되는 논리적 데이터 모임이다. 데이터베이스 테이블, 시스템 내부 파일, 클래스 등이 될 수 있고, 금번 프로젝트에서 생성하여 관리하는 데이터가 해당한다.

EIF는 측정대상 애플리케이션에서는 참조만 하고 다른 애플리케이션에서 유지되는 파일이다. 다른 프로젝트에서 생성했지만 이번 프로젝트에서 참조하는 데이터가 이에 해당한다. 핵심 기준은 물리적으로 내 DB에 있는지가 아니라 이 시스템이 등록·수정·삭제 책임을 지는가이다.

### 트랜잭션 기능: EI, EO, EQ

트랜잭션 기능은 입력, 출력, 조회를 측정한다. EI는 외부입력으로 데이터베이스에 데이터를 등록하거나 수정·삭제하는 기능이다. EO는 외부출력으로 계산 로직을 거쳐 데이터나 제어정보를 사용자에게 보여주는 기능이다. 수학적 계산 로직이 하나 이상 존재하고 그에 따른 파생 데이터가 있으면 EO다. EQ는 외부조회로, 로직이 필요 없고 DB에 존재하는 데이터를 찾아 그대로 표시만 하는 기능이다.

함정 포인트: 화면이 조회 화면이라고 해서 무조건 EQ가 아니다. 학점 평균 산출, 월별 매출 보고서, 이자 계산서처럼 SUM, AVG, 수식, 알고리즘, 파생값이 있으면 EO다.

### Decision Tree와 SQL 판정

부록 A의 의사결정 흐름은 네 질문으로 정리된다. 첫째, 측정 대상이 데이터인가 처리인가. 데이터라면 둘째, 이 시스템에서 CUD 책임을 지는가. 책임지면 ILF, 참조만 하면 EIF다. 처리라면 셋째, 데이터가 시스템으로 들어오거나 ILF를 변경하는가. 그렇다면 EI다. 출력이라면 넷째, 계산·가공·유도 로직이 있는가. 있으면 EO, 없으면 EQ다.

SQL 기준으로 INSERT, UPDATE, DELETE, MERGE/UPSERT는 EI다. 단순 SELECT, ORDER BY, LIMIT/OFFSET, 단순 JOIN은 EQ다. COUNT, AVG, MAX, SUM, GROUP BY 집계, price×qty 같은 파생 컬럼은 EO다. 단순 정렬과 페이징은 계산이 아니다.

### UFP와 개발원가

데이터 기능점수 = ILF 개수×7.5 + EIF 개수×5.4 이다. 트랜잭션 기능점수 = EI 개수×4.0 + EO 개수×5.2 + EQ 개수×3.9 이다. 미조정기능점수 UFP는 데이터 기능점수와 트랜잭션 기능점수의 합이다. UFP는 단순히 기능적 요구사항에 대해서만 계산하며 성능, 품질, 운영환경 같은 비기능 요소는 제외한다.

보정 전 개발원가는 UFP×기능점수당 단가이다. 자료에서는 기능점수당 단가 605,784원이 사용된다. 보정 후 개발원가는 보정 전 개발원가×(규모보정계수×애플리케이션유형보정계수×언어보정계수×품질/특성보정계수)이다. 학사관리 예에서는 UFP 158, 보정 전 원가 95,713,872원, 종합 보정계수 0.52, 보정 후 개발원가 49,770,823원이 제시된다.

### 시험 함정 포인트

- **정규기능점수법**: 유형별 복잡도 계산을 적용한다.
- **간이기능점수법**: 초기 단계 산정에 적합하다.
- **개발 프로젝트 기능점수**: 개발규모 측정이다.
- **개선 프로젝트 기능점수**: 변경규모 측정이다.
- **애플리케이션 기능점수**: 개발+개선 기능점수까지 포함한다.
- **측정범위**: 전체 시스템 또는 일부 시스템이 될 수 있다.
- **애플리케이션 경계**: 사용자 관점으로 구분해야 한다.
- **ILF**: CUD 책임이 이 시스템에 있다.

## 제5장 일정계획, WBS, CPM, 위험분석, 기본 용어

자료: 3장_계획의 이해.pdf, 3장_비용산정기법.pdf, 각종용어정리_노션.pdf, 14주차_하위설계_노션.pdf

### 주요 학습 주제
- 일정계획은 작업순서, 소작업 개발기간, 필요한 자원을 정하는 활동이다.
- WBS는 프로젝트 목표 달성에 필요한 활동과 업무를 계층 구조로 세분화한다.
- PERT/CPM은 작업순서와 소요기간을 네트워크 그래프로 표현하고 임계경로를 찾는다.
- 위험관리는 위험요소 식별, 위험분석, 위험계획 수립, 위험감시로 진행된다.
- 시스템, 방법, 프레임워크, 아키텍처, 모듈은 수준과 역할이 다르다.

### 일정계획

일정계획은 작업순서 결정, 소작업의 개발기간, 순서, 필요한 자원 등을 계획하는 활동이다. 학사관리 애플리케이션 예에서는 사용자정보관리, 교과과정관리, 수업관리, 수강관리, 성적관리, 테스트와 같은 개발 순서가 제시된다. 필요한 자원으로는 개발툴, 개발용 PC, 개발공간 등이 제시된다.

### WBS

작업분할구조도 WBS는 Work Breakdown Structure의 약자로, 프로젝트 목표를 달성하기 위해 필요한 활동과 업무를 세분화하는 작업이다. 프로젝트 구성요소들을 계층구조로 분류하고, 전체 범위를 정의하며, 프로젝트 작업을 세분화한다. 계층구조에서 최하위 항목을 작업패키지라고 한다.

WBS의 목적과 용도는 사용자와 개발자 간 의사소통 도구, 업무 내역 가시화, 관리 용이성, 팀원 책임과 역할 명확화, 필요 인력과 일정계획의 기초, 개발비 산정 기초, 성과 측정 및 조정 기준선이다.

### PERT/CPM과 네트워크 차트

PERT/CPM은 WBS의 작업순서와 소요기간을 네트워크 형태의 그래프로 표현한다. 이를 통해 어떤 작업이 중요한지, 여유가 있는 작업은 무엇인지, 프로젝트 완료 최소기간은 얼마인지, 완료기간을 맞추기 위해 각 작업을 언제 시작하고 완료해야 하는지 판단한다. 또한 일정 지연을 막기 위해 어떤 작업을 중점 관리해야 하는지, 전체 기간 단축을 위해 어떤 작업을 줄이는 것이 경제적인지도 분석한다.

### CPM 계산 요소

ES는 Earliest Start Time으로 가능한 가장 빠른 시작시간이다. 선행작업이 완료되었을 때 해당 작업을 시작할 수 있는 가장 빠른 시점이며 앞에서 끝 방향으로 계산한다. EF는 Earliest Finish Time으로 가장 빠른 완료시간이며 ES+작업소요시간이다. LS는 Latest Start Time으로 전체 일정 지연 없이 늦어도 시작해야 하는 시간이며 뒤에서부터 계산한다. LF는 Latest Finish Time으로 가장 늦게 끝낼 수 있는 시간이며 LS+작업소요시간으로 설명된다. ST는 Slack Time으로 여유시간이며 자료에서는 ES-LS로 제시된다. 임계경로는 여유시간이 없는 경로다.

함정 포인트: ES/EF는 앞에서 뒤로, LS/LF는 뒤에서 앞으로 계산한다. 임계경로는 작업 수가 많은 경로가 아니라 여유시간이 없는 경로다.

### 위험분석과 위험관리

위험분석은 프로젝트에서 발생할 수 있는 위험을 다루는 활동이다. 위험관리 절차는 위험요소 식별, 위험분석, 위험계획 수립, 위험감시로 이어진다. 위험분석에서는 위험요소가 발생할 가능성과 영향력을 판단한다. 과거 프로젝트 데이터와 위험 분석 경험이 많은 개발자에 의존해 판단한다.

위험 발생 가능성 척도는 매우 낮음(<10%), 낮음(10~25%), 보통(25~50%), 높음(50~75%), 매우 높음(>75%)으로 제시된다. 위험 발생확률은 상 80% 이상, 중 30~80%, 하 30% 미만으로도 분류된다. 영향력은 재앙, 심각함, 해결가능함, 경미함 등으로 나뉘며, 비용과 일정 기준으로는 상 20% 이상 초과, 중 5~20%, 하 5% 이하로 제시된다.

### 기본 용어

시스템은 상호 작용하는 여러 구성요소들이 특정 목적을 위해 조직된 전체 구조이며 입력-처리-출력 과정을 통해 목적을 달성한다. 방법/기법은 문제 해결이나 목표 달성을 위한 절차적 접근 방식 또는 규칙의 집합이다. 프레임워크는 특정 목적을 달성하기 위해 사용할 수 있는 재사용 가능한 구조와 기본 코드의 집합이며, 개발자가 코드를 추가하거나 재정의할 수 있고 제어 흐름이 내장된다. 아키텍처는 시스템 구성 요소들의 전체 구조와 상호작용 방식에 대한 설계 청사진이다. 모듈은 시스템이나 소프트웨어 내에서 독립적으로 기능을 수행하는 단위 구성요소이며 저결합과 고응집이 이상적이다.

### 시험 함정 포인트

- **일정계획**: 개발 순서와 자원 배분을 다룬다.
- **WBS**: 범위 정의와 일정·비용 산정 기초다.
- **작업패키지**: 실제 관리 가능한 작업 단위다.
- **PERT/CPM**: 중점 관리 작업과 최소 완료기간을 파악한다.
- **ES**: 앞에서 끝 방향으로 계산한다.
- **EF**: EF=ES+작업소요시간이다.
- **LS**: 뒤에서부터 계산한다.
- **LF**: LS+작업소요시간으로 설명된다.

## 제6장 상위·하위설계와 GoF 디자인패턴

자료: 상위설계2.pdf, 14주차_하위설계_노션.pdf, 디자인패턴.html, 각종용어정리_노션.pdf

### 주요 학습 주제
- 디자인 패턴은 객체지향 설계에서 반복되는 문제를 해결하기 위해 검증된 설계 경험을 추상화·일반화한 템플릿이다.
- GoF 패턴은 생성, 구조, 행위 패턴으로 구분된다.
- 생성 패턴은 객체 생성과 변경이 시스템 전체에 미치는 영향을 줄인다.
- 구조 패턴은 클래스와 객체를 합성해 더 큰 구조를 만든다.
- 행위 패턴은 객체 간 상호작용, 책임 분산, 알고리즘 교체와 관련된다.
- 하위설계 자료는 Python 예제와 팀 프로젝트 적용, WBS, 협업 관점에서 패턴을 설명한다.

### 디자인 패턴의 정의와 장단점

디자인 패턴은 객체지향 개념에 따른 설계 중 재사용할 경우 유용한 설계를 정립한 것이다. 설계 사례를 분석하여 서로 비슷한 문제를 해결하기 위한 설계로 분류하고, 문제 유형별 가장 적합한 설계를 일반화한다. 빈번하게 사용되는 설계 형태를 정형화하여 템플릿으로 만들고, 경험과 설계 지식을 검증, 추상화, 일반화한 템플릿이다.

장점은 설계 지식과 노하우를 문제 유형별로 구체화할 수 있고, 개발자 간 의사소통이 원활해지며, 구조 파악이 쉬워지고, 재사용을 통해 개발 시간이 단축되며, 설계 변경 요청에 유연하게 대처할 수 있다는 점이다. 단점은 객체지향 설계·구현 위주이고 초기 투자비용 부담이 있다는 점이다.

### GoF 3분류

생성 패턴은 객체를 생성하는 것과 관련된 패턴이다. 객체의 생성과 변경이 전체 시스템에 미치는 영향을 최소화하여 유연성을 높이고 코드 유지보수를 쉽게 한다. 포함 패턴은 Factory Method, Singleton, Prototype, Builder, Abstract Factory이다.

구조 패턴은 자료구조나 인터페이스 구조 등 프로그램 구조 설계에 활용되는 패턴이다. 클래스나 객체들의 합성을 통해 더 큰 구조로 만들 수 있게 하며, 복잡한 큰 시스템에서 새로운 기능을 가진 복합 객체를 효율적으로 생성할 수 있게 한다. 포함 패턴은 Adapter, Composite, Bridge, Decorator, Facade, Flyweight, Proxy이다.

행위 패턴은 반복적으로 사용되는 객체들의 상호작용을 패턴화한다. 클래스나 객체들이 상호작용하는 방법과 책임을 분산하는 방법, 알고리즘 대체, 메시지 교환과 관련된다. 포함 패턴은 Iterator, Observer, Strategy, Template Method, Visitor, Chain of Responsibility, Command, Mediator, State, Memento, Interpreter 등이다.

### 생성 패턴

Factory Method는 객체 생성 시점은 알지만 어떤 구체 클래스 객체를 생성해야 할지 모를 때 객체 생성을 하위 클래스에 위임한다. Singleton은 특정 클래스 객체가 오직 하나만 존재하도록 보장한다. Prototype은 new보다 clone을 이용해 기존 객체를 복사하고 일부만 바꿔 인스턴스를 생성한다. Builder는 복잡한 인스턴스를 조립하여 만들며 생성 과정과 표현 방법을 분리한다. Abstract Factory는 관련 있는 여러 제품군을 추상 인터페이스로 생성하여 일관된 조합을 제공한다.

함정 포인트: Factory Method는 제품 하나의 생성을 하위 클래스에 맡기는 관점이고, Abstract Factory는 관련 객체 집합 또는 제품군을 함께 생성하는 관점이다.

### 구조 패턴

Composite는 부분-전체 계층구조에서 단일 객체와 복합 객체를 동일하게 다룬다. Adapter는 호환되지 않는 기존 클래스의 인터페이스를 변환하여 재사용하게 한다. Bridge는 기능의 클래스 계층과 구현의 클래스 계층을 연결하고 분리하여 각각 독립적으로 확장하게 한다. Decorator는 기존 객체를 감싸 기능을 동적으로 추가하며 상속의 대안으로 사용된다. Facade는 복잡한 서브시스템 앞에 대표 창구 역할의 단순 인터페이스를 제공한다. Flyweight는 동일한 것은 공유해 객체 생성을 줄이고 메모리를 절약한다. Proxy는 실제 객체를 대신하여 지연 로딩, 접근 제어, 캐싱 등의 부가 처리를 수행한다.

함정 포인트: Adapter는 맞지 않는 인터페이스를 끼워 맞추는 것이고, Bridge는 추상과 구현을 독립적으로 확장하기 위해 분리하는 것이다. Decorator와 Proxy는 모두 감싸는 구조지만 Decorator는 기능 추가가 목적이고 Proxy는 접근 제어·지연 로딩·대리 처리가 목적이다.

### 행위 패턴

Iterator는 컬렉션 내부 구조를 노출하지 않고 동일한 인터페이스로 순회하게 한다. Observer는 어떤 클래스에 변화가 생겼을 때 이를 감지하여 다른 클래스에 통보한다. Strategy는 알고리즘 군을 정의하고 각각 클래스로 캡슐화하여 필요할 때 교환해서 사용하게 한다. Template Method는 상위 클래스에서 알고리즘의 틀을 정하고 구체 내용은 하위 클래스에서 결정하게 한다. Visitor는 객체 구조와 처리 기능을 분리하여 구조 변경 없이 기능을 추가하기 쉽게 한다.

Chain of Responsibility는 처리 책임이 연결되어 요청을 처리할 수 있는 객체까지 자동 전달한다. Command는 요청을 객체로 캡슐화하고 execute와 undo 같은 기능을 제공한다. Mediator는 객체들이 직접 얽히지 않고 중재자를 통해 상호작용하게 한다. State는 객체 상태를 클래스로 캡슐화하여 상태에 따라 동작이 달라지게 한다. Memento는 객체 상태를 저장하고 복원하여 undo 기능에 유용하다. Interpreter는 간단한 언어의 문법을 클래스로 정의하고 해석한다.

함정 포인트: Strategy와 State는 구조가 비슷하지만 Strategy는 외부가 알고리즘을 선택하여 교체하는 것이고, State는 내부 상태 변화에 따라 동작이 달라지는 것이다. Command의 undo는 명령 실행의 취소 구조이고, Memento는 상태 스냅샷 저장·복원 구조이다.

### 하위설계와 프로젝트 적용

하위설계 자료는 디자인 패턴을 Python 코드와 팀 프로젝트 적용 관점에서 설명한다. Factory Method는 보고서 포맷에 따라 PDF, Excel, Markdown 보고서 객체 생성을 분리하는 예시로 제시된다. Strategy는 정렬 알고리즘을 교체하는 구조로, Observer는 데이터 갱신 시 UI와 로거가 동시에 반응하는 구조로 제시된다.

ReportGen 프로젝트 예시는 Factory Method, Strategy, Observer를 적용한다. 팀장/설계자는 Factory Method와 전체 설계, 백엔드 개발자는 Strategy와 정렬 알고리즘, 프론트 개발자는 Observer와 UI 상태 반영, 테스트/품질 담당은 통합 테스트와 문서화를 맡는다. WBS와 GitHub Flow, 코드리뷰, 역할 분담이 함께 설명되어 디자인 패턴이 단순 코드 재사용이 아니라 협업 구조와 의사소통 표준으로도 기능함을 보여준다.

### 시험 함정 포인트

- **디자인 패턴**: 설계 지식의 재사용과 의사소통에 중요하다.
- **생성 패턴**: 생성과 변경의 영향을 줄인다.
- **구조 패턴**: 복잡한 구조를 효율적으로 설계한다.
- **행위 패턴**: 메시지 교환과 알고리즘 교체에 관련된다.
- **Factory Method**: 구체 객체 생성을 분리한다.
- **Singleton**: DB 연결, 설정 객체 같은 공유 자원에 사용된다.
- **Prototype**: 복잡한 객체 생성 비용을 줄인다.
- **Builder**: 선택 필드가 많은 객체 구성에 유용하다.
